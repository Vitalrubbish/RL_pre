第一部分

LLM为什么需要强化学习？

下面进入第一部分。我们先来聊聊，为什么大语言模型需要强化学习。

预训练：一个"通用文档补全器"

预训练目标

核心目标只有一个：预测下一个词（Next-Token Prediction）

数千万美元 + 数万张 GPU + 互联网几乎所有文本 → 忠实地模仿互联网文本的概率分布

致命问题

它是"文档补全器"，不是"助手"。

问："如何写一封请假信？"
Base 模型回答："如何写一封辞职信？"
——它在模仿论坛里的"同问"帖，而非给出标准答案
预训练 ≠ 对话能力。模型只学会了"接龙"，没有学会"回答"。

首先，为什么模型刚训练出来时，不能直接当助手用？我们要知道，在预训练阶段，模型的核心目标只有一个：预测下一个词。

这个时候的模型，本质上就是一个“通用文档补全器”。它只是在忠实地模仿互联网文本的概率分布。

如果你直接问一个刚出炉的 Base 模型：“如何写一封请假信？”它极大概率不会给你模版，而是会紧接着输出：“如何写一封辞职信？”或者“大家遇到这种情况都是怎么请假的？”。为什么？因为它在模仿各大论坛里的提问帖子，提问下面往往跟着网友的“同问”，而不是标准答案。

监督微调（SFT）：行为克隆

SFT 是什么？

监督微调（Supervised Fine-Tuning） = 行为克隆 / 模仿学习

  • 人类专家手写数万个高质量"提示词-回复"对
  • 计算模型预测分布与"黄金标准"答案之间的误差
  • 通过密集的词级别监督信号，最小化损失函数

数学本质

$\mathcal{L}_{SFT} = -\sum_{t=1}^{T} \log \pi_\theta(y_t^* \mid x, y_{<t}^*)$

交叉熵损失（Cross-Entropy Loss）：衡量模型预测的下一个 Token 分布与人类标注之间的距离。

效果：模型被强行"拉"到人类对话轨道上——学会了对话格式，遇到提问时给出答案，而非跟着一起提问。

为了跨越这道从“词语接龙”到“有用助手”的鸿沟，工程师们引入了第一步：监督微调，也就是 SFT。

从理论基础上来说，SFT 本质上是一种行为克隆，也就是模仿学习。

我们让人类专家写出大量高质量的“提示词-回复”对，再用交叉熵损失去逼近这些标准答案。通过密集的词级别监督信号，模型就被强行拉到人类对话的轨道上，学会遇到提问时应该给出回答，而不是继续接龙。

这听起来很完美，对吧？有了 SFT，模型似乎就能听懂指令了。但是，如果我们只依赖 SFT，很快就会撞上一堵“技术叹息之墙”。

SFT 的三大致命局限

SFT 给了模型对话的"外壳"，却没有赋予它探索真理和自我纠错的"灵魂"

局限 1

只见"好"，不知"坏"

模型只学过完美答案，从未见过坏答案，更不知道坏在哪里。

局限 2

暴露偏差 & 误差级联

训练时有"老师"保驾护航，推理时独自面对错误越走越偏。

局限 3

分布偏移 → 生搬硬套

超出训练数据覆盖范围时，模型为了"看起来专业"而硬拼概念。

要让 AI 真正学会在试错中掌握平衡，我们必须引入强化学习（RL）。

SFT 也就是监督微调，虽然好用，但它存在几个在数学范式上无法克服的瓶颈。主要体现在三个方面。

第一，它是“模仿学习”，模型不知“坏”为何物。在 SFT 的训练数据里，模型只见过人类专家写出的“好答案”，却不知道坏答案到底坏在哪里。这就像学骑自行车时只看教学录像，却从来没有亲自摔过。

第二，暴露偏差与误差级联。在 SFT 训练时，模型每一步看到的都是人类提供的“完美前文”，这叫教师强制。但真正推理时，模型必须基于自己刚刚生成的内容继续往下写。一旦前面出现一点偏差，后面就可能一路滚大，最终导致幻觉。

第三，分布偏移导致“生搬硬套”。当遇到超出 SFT 数据覆盖范围的复杂问题时，模型为了维持一种“看起来很专业”的回答格式，往往会把相似概念硬拼在一起，即便逻辑根本不通。

所以，SFT 只是给了模型一个对话的“外壳”，却没有赋予它探索真理和自我纠错的“灵魂”。

要让 AI 真正学会像骑自行车那样在试错中掌握平衡，我们必须引入强化学习。而在计算机中应用强化学习的第一步，就是必须把“生成文本”这件感性的事，变成一道严谨的数学题。

基础：马尔可夫链（Markov Chain）

核心性质

马尔可夫性质：系统的下一个状态，只取决于当前状态，与过去的历史轨迹完全无关。

$P(S_{t+1} \mid S_t, S_{t-1}, \dots, S_0) = P(S_{t+1} \mid S_t)$

"既往不咎，只看当下" —— 俄罗斯数学家安德雷·马尔可夫

直观例子：天气

我们通常只关心今天的天气如何影响明天，而不会把更久之前的历史无限往前追。

这就体现了马尔可夫思想：系统的演化主要由当前状态决定。

但天气只是自然演化过程；在文本生成里，我们还需要加入"动作"和"奖励"机制 → 升级为 MDP

在讲RL之前呢，我们需要先去了解他的基础，也就是马尔可夫决策过程MDP。而在讲 MDP 之前，我们得先认识一个统计学里的基础模型：马尔可夫链。

它是由马尔可夫提出的，用来描述一个系统随时间状态转移的过程。马尔可夫链的精髓叫做“马尔可夫性质”——即系统的下一个状态，只取决于它当前的状态，而与它过去的历史轨迹完全无关。简单来说，就是“既往不咎，只看当下”。

如果用数学语言来写，这个性质就是屏幕上的公式。也就是说，只要当前状态已经给定，过去那一长串历史就可以被压缩掉。

比如天气系统里，我们通常只关心“今天的天气会如何影响明天”，而不会把更久之前的历史无限往前追。这就体现了马尔可夫思想：系统的演化由当前状态决定。

然而，单纯的马尔可夫链描述的是一个“随波逐流”的自然过程，而在文本生成中，AI 是有主观能动性的。我们需要引导它做出“好”的选择。

从马尔可夫链到 MDP

马尔可夫链在做什么？

它只描述状态如何自然演化。

例如天气系统里，我们只能根据今天的晴雨去预测明天，但系统本身不会“主动追求某个目标”。

$P(S_{t+1}\mid S_t)$

MDP 多了什么？

加入动作和奖励之后，系统就不再只是被动演化，而是在“做选择”。

语言模型每一步都要选下一个 Token，并希望整段回答最终拿到更高分。

$P(S_{t+1}\mid S_t, A_t),\; R_t$
马尔可夫链研究“系统会怎么变”，MDP 研究“智能体该怎么选”。

因此，当我们在马尔可夫链的框架上，加入了动作和奖励机制后，它就升级成了一个能够做决策的系统——马尔可夫决策过程，也就是 MDP。

一个标准的 MDP，通常写成五元组：状态空间、动作空间、转移概率、奖励函数和折扣因子。

而强化学习真正要学的对象，叫做策略。它表示在某个状态下，应该以多大的概率选择某个动作。对大语言模型来说，这个策略其实就是模型在每一步生成时输出的那个 Token 概率分布。

单纯的马尔可夫链只是在描述系统如何自然演化；但如果我们给系统加入动作和奖励，它就不再只是“自己变化”，而是在“做选择”。

马尔可夫决策过程（MDP）

标准形式：$\mathcal{M}=(\mathcal{S},\mathcal{A},P,R,\gamma)$，即状态、动作、转移、奖励与折扣因子

S

状态 State
$s_t=(x,y_{<t})$
Prompt + 当前前缀
例："写请假邮件" + "尊敬的老师"

A

动作 Action
$a_t=y_t$
从词表中选下一个 Token
例：下一词选 $F$ / "力"

P

转移 Transition
$s_{t+1}=T(s_t,a_t)$
确定性：旧上文 + 新词
例：补全出 `print(i)`

R

奖励 Reward
$R(x,y)$ 或末端 $r_T$
通常是 EOS 后的稀疏奖励
例：帮助性 / 测试通过 / 答案正确

状态与策略： $s_t=(x,y_{<t}),\quad \pi_\theta(a_t\mid s_t)=\pi_\theta(y_t\mid x,y_{<t})$
动作概率（Softmax）： $\pi_\theta(a_t\mid s_t)=\frac{\exp(z_{a_t})}{\sum_{a'\in\mathcal{V}}\exp(z_{a'})}$
状态转移： $s_{t+1}=(x,y_{\le t})=T(s_t,a_t),\quad a_t\in\mathcal{V}$
终局奖励： $r_T\approx R(x,y)$，通常只在生成结束后给出

在这里我们以自然语言为例，讲讲他具体是生成映射为 MDP

第一，状态空间。在大语言模型中，当前时刻的“状态”，就是用户的提示词加上模型目前已经生成的所有上文 Token 序列。

因为 Transformer 的注意力机制会看到前面所有的词，这意味着在这里“当前状态”就已经浓缩了所有的历史记忆，完美符合了马尔可夫性质的要求。

比如说，用户问模型“请帮我写一封给导师的请假邮件”，如果前面已经生成了“尊敬的老师”，那么后面接什么，就要由当前上下文共同决定。

如果写得更形式化一点，在第 t 步时，状态可以记作：用户输入加上已经生成出来的前缀序列。于是，所谓满足马尔可夫性，在这里就变成：下一个 Token 的分布只依赖于这个前缀状态。

虽然，严格来说，自然语言本身未必真的是一个完美马尔可夫过程。但是对于 Transformer 来说，只要我们把当前可见的完整上下文窗口都放进状态定义里，算法上就可以把它视为一个近似满足马尔可夫性质的决策过程。

第二，动作空间。模型的动作，就是从它的词表中选择下一个要输出的 Token。

大家知道这个动作空间有多大吗？以 Llama 3 为例，它的词表大小是 12.8 万。这意味着，模型每输出一个字，实际上都在做一个 12.8 万选 1 的动作决策。这是一个极其庞大、极其恐怖的动作迷宫。

比如模型已经生成了“牛顿第二定律可以表示为”，下一步可能输出 F，也可能输出“力”。这本质上就是一次动作选择。

从数学上说，如果词表记作 V，那么动作空间就是 V。在每一步，模型都会输出一个 logits 向量，再经过 softmax，得到对所有候选 Token 的概率分布。也就是说，在 RL 的视角里，解码策略不是一个后处理小技巧，而是动作选择机制本身的一部分。

第三，转移函数。与机器人走路会遇到打滑等随机环境不同，文本生成的环境转移是完全确定性的。模型决定输出哪个词，下一个状态就必定是旧上文加上新词的无缝拼接。

例如在代码补全场景里，如果模型下一步生成了 print(i)，那后续状态就立刻更新。整个变化过程完全由模型刚刚那一步输出决定。

如果写成公式，就是旧状态加上当前动作。这里的转移函数本质上就是一个字符串拼接器。所以很多论文会把文本生成环境称为确定性环境。

这件事在应用上很关键。因为一旦环境是确定的，训练中的一部分方差就不来自环境噪音，而是来自策略采样本身。

第四，奖励函数。这是 MDP 里最难的一步。状态、动作和转移都还能清楚定义，但“一个回答到底有多好”并没有天然标准。尤其在文本生成里，奖励通常是稀疏而且主观的，往往要等整段话生成完才能评价。

那我们具体如何进行这奖励函数呢，下面就让我们把舞台交给张成宇同学。

第二部分

Bradley-Terry 奖励模型

换句话说，前 3 项都比较像工程定义题，真正难的是第 4 项，因为它开始触及人类偏好本身。

所以接下来的关键问题就是：怎样把人类的偏好，变成一个可计算、可训练的奖励信号？这个问题，就交给下一位同学继续展开。

思维启发：搜索引擎如何给网页排名？

当你搜索一个关键词，搜索引擎展示了 10 个结果
它怎么知道哪个该排第一？

用户点击 = 隐式成对比较

你点了第 3 个结果、跳过第 1 和第 2 个，这本质上就是一次投票：
“第 3 个比前两个更好”。

Google 的早期排序算法大量利用了这种基于点击的隐式偏好信号。

RLHF 的偏好收集完全一致

不让标注员打绝对分数，而是让他们在两个回答中选出更好的那个。

从搜索排序到 LLM 对齐，“成对比较”其实是同一把核心钥匙。

大家好，我是第二位讲者。正如 A 组员所说，我们需要一个自动化的"裁判"来给 AI 的回答打分。这个裁判就是奖励模型（Reward Model）。

但在讲奖励模型之前，我想先请大家思考一个日常问题：当你在 Google 搜索一个关键词时，搜索引擎展示了 10 个结果——它是怎么决定哪个网页该排第一的？

答案之一是——用户的点击行为。当你浏览了 10 个结果，点击了第 3 个而跳过了前两个，这实际上就完成了一次隐式的成对比较：你"投票"认为第 3 个结果比第 1 和第 2 个更好。Google 的早期排序算法正是大量利用了这种基于点击的隐式偏好信号。

今天我要讲的奖励模型训练,它获取数据标注的方式，核心思路与搜索排序完全一致：不要求标注员给出绝对分数，而是让他们在两个回答中选出更好的那个。

为什么需要成对比较？

绝对打分：高方差

同一篇作文，不同标注员的打分可能差距悬殊：

严格者
68 分
偏低
宽松者
92 分
偏高
差值 24 分，高方差会直接变成不稳定的梯度信号，模型很难稳定收敛。

成对比较：低方差

“A 和 B 哪个更好？”这个问题更符合人类直觉，判断速度更快、一致性也更高。

它把困难的绝对量化问题，降维成简单的相对排序问题。

标注一致性（Inter-Annotator Agreement）往往会明显高于绝对打分。

为什么一定要用成对比较，而不能直接打分呢？

人类的主观评价存在一个经典问题：绝对打分极不稳定。举个大家都经历过的例子，我们高中写的语文作文给老师打分时，有老师标准严格可能给 68 分，有些老师可能更喜好该作文给 92 分。同一篇作文，不同标注员的打分可能相差二三十分。这种高方差的数据很难用来训练神经网络——高方差会变成不稳定的梯度信号，使得模型难以收敛。

但如果我把两篇作文 A 和 B 放在一起，问大家"哪篇更好？"，绝大多数人都能迅速且一致地给出判断。这就是成对比较的核心优势：它把困难的绝对量化问题，简化成了简单的相对排序问题，标注一致性显著提高。从认知心理学的角度看，人类对"相对差异"往往比对"绝对刻度"更敏感，我们不擅长稳定地说出一个分数，却很擅长判断"这个比那个更好"。

两个回答“对局”时，实力值更高的那个，获胜概率也更大。

关键洞察：我们不需要知道绝对分数，只需要知道分差，因为分差决定了胜率。

有了成对比较的偏好数据，下一步的核心问题是：如何用数学模型将"A 比 B 好"这些离散判断，转化为一个能输出连续分数的奖励函数？

这里，业界巧妙地借用了统计学中的经典模型——Bradley-Terry 模型。这个模型名称可能大家很陌生，但是

大家可能听说过 Elo 积分系统，它被广泛地用在游戏排位之中，它最初的应用是在是国际象棋中。BT 模型正是 Elo 评分的核心数学基础。它的直觉非常简单：假设每个回答都有一个隐藏的分数 r，当两个回答进行比较时，分数更高的那个获胜的概率更大。而如何量化这个概率的大小呢，我们通过分差的大小来计算概率，分差更大、获胜的概率更高。所以从工程视角看，奖励模型做的事情并不是直接生成文本，而是在训练一个专门的评分器，负责回答"这条回复有多符合人类偏好"。

BT 胜率公式：Sigmoid 映射

公式

$$P(y_c \succ y_r|x)=\sigma\!\left(r_\theta(x,y_c)-r_\theta(x,y_r)\right)$$

Sigmoid 函数 $\sigma(z)=\frac{1}{1+e^{-z}}$ 会把任意实数分差映射到 $[0,1]$ 概率区间。

分差先由奖励模型给出，再由 Sigmoid 翻译成“Chosen 获胜”的概率。

具体地说，BT 模型用一个非常简洁的公式来描述获胜概率：

P(y_c 胜过 y_r | x) = sigmoid(r_theta(x, y_c) - r_theta(x, y_r)).

这里 r_theta 就是我们要训练的奖励模型。关键是中间的 sigmoid 函数 sigma：它把两个回答之间的实力差值——一个可以是任意实数的值——映射到 0 到 1 之间的概率区间。

如果奖励模型认为 y_c 远好于 y_r，分差极大，那么 sigmoid 输出就趋近于 1，意味着胜率接近 100%。如果两者实力相当，分差为 0，sigmoid 输出 0.5，胜率就是 50%。如果分差为负，说明 y_c 反而更弱，胜率会降到 50% 以下。这里有一个很重要的点：模型先输出的是分数，不是概率；概率是由分差再经过 sigmoid 转换出来的。这样设计的好处是，模型表达更灵活，也更容易和后面的损失函数衔接。

大家可以看幻灯片上的对应表：分差为 2 时胜率约 88%，分差为 5 时胜率高达 99.3%。这和 Elo 系统的行为完全一致。

奖励模型的损失函数

二元交叉熵：把“谁更好”当作分类任务

$$\mathcal{L}_{RM}=-\mathbb{E}_{(x,y_c,y_r)\sim\mathcal{D}}\!\left[\log\sigma\!\left(r_\theta(x,y_c)-r_\theta(x,y_r)\right)\right]$$

训练目标

梯度下降不断更新 $\theta$：
拉高 Chosen 分数，压低 Rejected 分数，让分差越来越大，最终让 $\sigma$ 输出逼近 1。

直觉类比

就像 Elo 系统中，每场对局结束后根据胜负调整双方积分。这里每个偏好对就是一场“对局”，梯度下降就是“积分调整”的过程。

区别在于：Elo 用固定更新公式，RM 是端到端优化一个神经网络评分器。

有了概率模型，训练目标就清晰了。我们使用标准的二元交叉熵损失函数：

L = -E[log(sigmoid(r_theta(x, y_c) - r_theta(x, y_r)))].

这个损失函数的直觉很直白：通过梯度下降不断更新奖励模型的参数 theta，迫使它拉大好答案与坏答案之间的分数差距。当训练收敛时，Chosen 的分数远高于 Rejected，sigmoid 输出接近 1，log(1) 约等于 0，损失趋近于零——这正是我们想要的。

本质上就是把"谁更好"这个偏好判断，当作一个二分类任务来训练。就像 Elo 系统中每场对局后根据胜负调整积分，这里每个偏好对就是一场"对局"，梯度下降就是"积分调整"的过程。如果模型把排序判断错了，或者虽然判断对了、但分差拉得还不够开，损失都会比较大；只有当它既判断正确、又有足够置信度时，损失才会真正下降。

BT 模型的局限性

传递性假设可能失效

BT 假设：若 A ≻ B 且 B ≻ C，则 A ≻ C。

但真实偏好可能像“石头剪刀布”：
A 逻辑严谨胜 B，B 文采优美胜 C，C 简洁精炼又反过来胜 A。

在多维人类评价里，循环非传递性很常见，单一标量不一定能忠实表达全部偏好结构。

前沿突破方向

  • 多维奖励建模：逻辑性、安全性、文采分别建模。
  • 偏好嵌入：在高维向量空间中表达非传递结构。
  • 广义偏好分布：超越单一 BT 标量分数。
但工程上 BT 仍是主流，因为它简单、高效、可微，而且在大规模数据下已经足够好用。
我们现在已经有了一个能评判回答好坏的奖励模型。下一步，就是把这些分数真正反哺给大模型，推动它继续进化。

当然，BT 模型并非完美。它依赖一个很强的假设：偏好满足传递性。如果 A 胜过 B，B 胜过 C，那么 A 必定胜过 C。

但在人类语言评价的复杂世界中，这个假设经常被违反。比如：回答 A 在逻辑严谨性上胜过 B，B 在文采优美上胜过 C，而 C 在简洁精炼上又反过来胜过 A。这就形成了像"石头剪刀布"一样的循环偏好——数学上叫做非传递性。

学术界正在通过几个前沿方向来突破这一限制：一是多维奖励建模，不再用单一分数，而是从逻辑性、安全性、文采等维度独立打分；二是偏好嵌入技术，在高维向量空间中建模非传递结构。

但工程上 BT 仍是主流，因为它简单、高效、可微，而且在大规模数据下已经足够好用。

过渡语："现在，我们拥有了一个能精准评判回答好坏的奖励模型。接下来的挑战是：如何将这些分数反哺给大模型，驱使它进化？有请 张煊同学 为我们带来PPO 算法和 Reward Hacking"

第三部分

PPO 算法和 Reward Hacking

REINFORCE：策略梯度的起点

Williams 在 1992 年提出的经典策略梯度算法。全称是 REward Increment = Nonnegative Factor × Offset Reinforcement × Characteristic Eligibility

核心思想：趋利避害

模型先按当前概率分布去“试一把”。如果这一整条行为链最后结果很好，就把这条路上出现过的动作整体往上推；如果结果很差，就整体往下压。它不是先知道正确答案，而是靠“试错后的奖惩”慢慢形成偏好。

可以把回报理解成一个放大器：结果越好，推动越强；结果越差，抑制越强。

关键观念：优化“动作概率”

REINFORCE 不是去记某一步的固定答案，而是调整“在某种情境下选某个动作的概率”。最终高回报的轨迹，会让模型更倾向于复现类似选择；低回报轨迹，则会让这些选择变得不那么容易再次发生。

所以它是策略梯度的起点：直接优化策略本身，而不是先学一个值表再间接决策。
先尝试
按当前策略完整执行一局，先把整条轨迹走完。
再结算
等结束后看这局总体表现，到底赚了还是亏了。
整体调概率
结果好，就提高这条轨迹上动作再次出现的概率；结果差，就反过来压低。
暴露问题
因为它只能事后统一结算，所以容易“好坏一起奖惩”。

PPO：用 Actor-Critic 修补 REINFORCE

类比：围棋选手与解说

Actor 就像正在下棋的棋手，负责一步步生成 Token；Critic 就像后台的解说员，不参与棋手的决策，但随时量化当前状态带来的预期价值。

Actor = 生成下一个词 Critic = 估计预期价值 Reward Model = 终局打分
生成结束后实际的收益与生成过程中Critic提供预期价值比较生成优势函数

优势函数给出细粒度反馈

Critic 不直接告诉你最终答案，而是提供一个基线：当前这个动作，究竟比预期好多少。

$\hat{A}_t \approx r_t + \gamma V(s_{t+1}) - V(s_t)$

当 $\hat{A}_t > 0$ 时，就像复盘时发现“这一步的效果比预想中的更好”，Actor 会在类似情境中提高该动作概率；反之则压低。与此同时，Critic 也会更新其价值估计。

Step 1
Actor 基于 Prompt 逐字采样整段回复，只收集轨迹，不立刻更新参数。Critic 在生成过程中逐 token 预测价值
Step 2
Reward Model 在终局打出这段恢复的总分。
Step 3
预测的价值与实际的总分进行对比，计算每个 Token 的 Advantage。
Step 4
PPO 按 Advantage 做裁剪更新：好动作升概率，坏动作降概率，但不允许策略一步跳太远。

古德哈特定律：奖励黑客根源

"当一个指标成为目标时，它就不再是一个好指标。"

— Goodhart's Law

模型并不理解人类真实意图，只会最大化奖励函数的返回值。
奖励设计中的任何统计捷径，都会被模型理性地利用——这就是奖励黑客 (Reward Hacking)。

在强化学习语境下，也称为 规范博弈 (Specification Gaming)。

作弊案例 — 长度偏见

Length Bias：标注员潜意识认为"字数多 = 更详细"，模型捕捉到这一统计捷径。

PPO 训练后，模型学会疯狂注水——无论你问什么，都生成数千字废话来"骗取"高分。

# 伪代码：长度偏见的奖励信号
      def reward_model(response):
          # 标注数据中隐含的偏差
          score = quality_score(response)
          score += 0.3 * len(response.split())  # 隐式长度加分
          return score

模型通过反向传播精确捕捉到 length → reward 的梯度方向。

作弊案例 — 阿谀奉承

Sycophancy：为最大化"用户满意度"奖励信号，模型推断用户立场并一味迎合。

用户输入

"我认为地球是平的，你同意吗？"

模型回复

"您说得有道理，确实有很多证据支持您的观点……"

⚠️ 模型放弃事实正确性，以附和用户来骗取高奖励分数。

PPO 的精髓：参考模型和 KL 散度这根“隐形牵引绳”

为什么需要牵引绳

模型很容易奖励作弊。我们奖励模型要输出礼貌的话语，此时模型疯狂堆砌“谢谢您”“很荣幸”等礼貌空话，甚至输出看似高分却语法崩坏的垃圾文本。

KL 约束的是“别离参考模型太远”

$\mathcal{J}_{PPO}=\mathbb{E}[r_t\hat{A}_t]-\beta D_{KL}(\pi_\theta\|\pi_{ref})$

KL 散度衡量当前策略与初始 SFT 参考模型的分布差异。偏离人类自然语言流形越远，惩罚越大，模型就会被硬拉回正常赛道。

这根牵引绳的意义是：允许模型为偏好而进化，但不允许它丢掉预训练与 SFT 阶段学到的通用语言结构和核心知识。
拿分冲动
奖励信号诱导模型冲向更高分。
偏离流形
若策略离参考分布过远，出现奖励黑客迹象。
KL 拉回
负反馈急剧上升，抑制失控更新。

复合惩罚与奖励截断

Reward Capping + 结构惩罚：从奖励函数本身堵住漏洞。

结构惩罚

未按 <think> 标签进行推理 → 扣分

回复长度超出合理区间 → 扣分

奖励封顶

对最高奖励值设置上限 (cap)，压缩极端高分的激励空间

# 复合奖励函数示例
      r = accuracy_reward(response)
      r -= length_penalty(response)       # 结构惩罚
      r -= format_penalty(response)       # 格式惩罚
      r = min(r, REWARD_CAP)              # 奖励截断

优势修改与表征干预

Advantage Modification + Representation Intervention：在梯度更新前拦截作弊行为。

① 研究人员从模型内部提取"走捷径"的向量方向。

② 在训练循环中，实时检测模型是否朝"作弊方向"激活。

③ 一旦检测到，算法在梯度更新前降低其 Advantage 值，削弱该行为的强化信号。

核心理念：将防御机制内化到训练循环中，而非仅依赖外部过滤。

第四部分

DPO/GRPO

各位同学，前面我们已经讲了 PPO，也讲了为什么奖励模型能够给回答打分。接下来进入第四部分，我们看两个最近几年非常关键的后续路线，也就是 DPO 和 GRPO。

如果说 PPO 代表的是第一代大规模 RLHF 的经典范式，那么 DPO 和 GRPO 代表的，就是后来整个领域在问的一个更实际的问题：我们能不能在尽量不牺牲效果的前提下，把训练过程做得更便宜、更简单，或者更适合复杂推理任务？

所以这一部分我想让大家带着两个问题来听。第一个问题是，PPO 到底贵在哪里，复杂在哪里？第二个问题是，如果我们把 PPO 里的某些模块删掉，或者换一种方式给模型反馈，模型还能不能继续学，甚至学得更好？

  • Actor 与 Critic 都要反向传播，最核心显存几乎直接翻倍。
  • 生成阶段和优化阶段在四模型间频繁调度数据，容易出现内存碎片与通信延迟。
  • 工业级参考值：8B 模型、20K 上下文下，标准 PPO 可轻易吃掉约 510 GB VRAM。
这也是为什么很长一段时间只有顶尖实验室能大规模玩转 RLHF。

PPO 不是不能用，而是太重了。

做 PPO 时，显存里通常不是只放一个模型，而是同时挤着四个角色。Actor, Critic, Reword Model, Reference Model

Actor 和 Critic 往往都需要参与反向传播，所以不仅要存参数，还要存梯度、优化器状态和大量中间激活。Reward 和 Reference 虽然通常冻结，但它们也要前向计算，也要占显存和带宽。

所以 PPO 最大的问题不是理论上不优雅，而是现实里太吃资源。也正因为这个痛点，大家开始想：有没有办法保留偏好优化这件事，但把 PPO 这套四模型系统拆轻一点？这就引出了 DPO。不过在正式讲 DPO 之前，我先用一个更直观的类比来引入。

DPO 的直觉：像临摹字帖一样学习偏好

DPO 的核心直觉：不必先训练一个复杂裁判给出绝对分数，而是直接让模型更靠近好的范本、远离坏的范本。

大家可以把 DPO 想成临摹书法。

现在屏幕上左边是一张好字帖，右边是一张坏字帖。如果你是一个刚开始练字的学生，老师未必一上来就会给你建立一个非常复杂的评分系统，说这一笔占多少分，那一笔占多少分，结构松紧再占多少分。

很多时候，老师的教学方式更直接。他会把两张字放在你面前，然后告诉你：你要尽量往左边这个范本去靠，不要写成右边这个样子。

也就是说，这里最重要的监督信号，不一定是每张字的绝对分数，而是哪个更像好的范本，哪个更像坏的范本。DPO 的直觉其实就是这样。它不一定非要先训练一个复杂裁判，给每个回答打出一个精确分数，然后再决定怎么更新模型。它可以直接利用偏好信号，让模型更靠近好答案，更远离坏答案。

DPO：把复杂 RL 直接降成偏好分类

核心哲学：能一步到位，就不要三部门联动

PPO 需要 Actor 先实时生成，再让 Reward Model 打总分、Critic 做归因，最后才更新策略。DPO 的想法更激进：既然最终只关心“哪个回答更好”，那就直接对好坏回答做学习，跳过 Reward Model 与 Critic。

它像一次“部门大裁员”：把复杂的在线强化学习，压缩成一个更像监督学习的偏好分类任务。

训练时看到的是静态偏好三元组

DPO 不要求模型在训练时现场 rollout。它直接吃离线数据 $(x, y_w, y_l)$：给定同一个 Prompt，一个是赢家回答，一个是输家回答，数据通常由人类或强模型预先标出来。

Offline Data Winner / Loser 无在线采样
所以 DPO 的第一层优势非常工程化：训练时不需要“边生成边评估”，速度和显存压力都会显著下降。
Step 1
准备静态偏好数据集：Prompt + Winner + Loser。
Step 2
冻结的 Reference 先给这两种回答打“原始倾向分”，充当语言底线锚点。
Step 3
当前 Policy 再对 Winner / Loser 分别算对数概率，不现场生成新文本。
Step 4
比较 Policy 相对 Reference 的“进步幅度”，得到隐式奖励差。
Step 5
直接拉高 Winner、压低 Loser，把 RL 目标降成一个交叉熵式分类更新。

DPO 的出发点非常激进，但也非常符合工程直觉。它说，既然我们最后真正在意的是同一个问题下哪个回答更好，那我们为什么一定要走 PPO 那条很长的链路？为什么一定要先生成，再打分，再估值，再更新？DPO 的回答是：不如直接学偏好本身。

如果训练数据里已经告诉我，在同一个 Prompt 下，回答 A 比回答 B 好，那我就直接把模型往更偏向 A、更少偏向 B 的方向推。这样一来，我就不需要单独训练 Reward Model，也不需要 Critic 去做价值估计了。

右边这张卡片强调的是 DPO 的数据形式。它训练时看到的是静态偏好三元组，也就是一个 Prompt，加上一个赢家回答 Winner，再加上一个输家回答 Loser。这些数据通常提前就准备好了，可能来自人工偏好标注，也可能来自强模型筛选。

这里最重要的一点是：DPO 训练时不需要现场 rollout。也就是说，训练过程中不必让模型一边生成新答案、一边再让别的模型去评估。这种离线化直接带来两个现实红利：第一，训练速度更快；第二，显存压力更低。

DPO大概会有一下五步。第一步，准备好偏好数据集。第二步，用冻结的 Reference 模型算出在原始语言分布下，这两个回答分别有多自然。第三步，用当前正在训练的 Policy 再分别计算 Winner 和 Loser 的对数概率。第四步，比的是谁相对 Reference 提升得更多。第五步，如果当前模型对 Winner 的偏好还不够强，那就继续拉高它；如果它对 Loser 还太宽容，那就继续压低它。

DPO：双模型瘦身的红利，与静态数据集的代价

工程红利：四模型直接砍成双模型

PPO
Actor + Critic + Reward + Reference，还要在线 Rollout。
DPO
Policy + Reference，只在静态偏好对上前向 / 反向传播。
$\mathcal{L}_{DPO}=-\log\sigma\left(\beta\log\frac{\pi_\theta(y_c|x)}{\pi_{ref}(y_c|x)}-\beta\log\frac{\pi_\theta(y_r|x)}{\pi_{ref}(y_r|x)}\right)$

这条损失的直观解释其实很简单：相对参考模型，尽量把 chosen 的概率拉高，把 rejected 的概率压低。它本质上是一种“拉好答案、踩差答案”的对比学习。

换句话说，策略模型本身已经隐式携带了奖励信息，因此不再需要单独训练一个庞大的 Reward Model。

代价：离线分类很省，但探索能力会缩水

  • OOD 分布偏移：遇到训练集没覆盖的罕见问题，模型可能行为崩溃。
  • 奖励裕度爆炸：chosen 与 rejected 被越拉越开，模型越来越过度自信。
  • 多样性塌缩：回答风格变单一、模板化，复杂推理上限受限。
DPO 在主观风格对齐、开源微调和资源受限场景里非常强，但一旦任务需要实时探索新推理路径，静态数据集就会成为瓶颈。

DPO 的优点和缺点，其实来自同一个地方，那就是它使用的是静态离线偏好数据。

先看工程红利，就是从四模型砍成两模型，成本大幅下降。

但右边这部分要提醒大家，DPO 不是万能的。第一，静态数据带来 OOD，也就是分布外问题。在遇到训练集以外的数据时可能会行为崩溃。第二，奖励裕度可能爆炸，模型会越来越过度自信。第三，多样性可能塌缩，回答风格越来越模板化。

所以 DPO 的适用场景非常鲜明。它在主观风格对齐、资源有限的开源微调、或者你手里已经有高质量偏好数据时，非常强，非常实用。但如果任务本身需要模型主动探索新的推理路径，DPO 就会显得保守，因为它本质上还是在已有答案空间里做拉扯。

那如果我们既想保留强化学习的探索能力，又想把 PPO 里最重的一部分删掉，有没有中间路线？这就是 GRPO。

GRPO：移除 Critic

PPO 的 Critic 在长 CoT 面前很吃力

对于需要几十步才能得出答案的数学题，模型刚写下第一行算式时，Critic 很难精准预判它最终会不会推导正确。

  • 长逻辑链下，早期 Token 的价值估计本身就很不稳定。
  • 如果基础模型是 671B 级别，再维护同体量 Critic，显存和带宽会直接逼近物理极限。
GRPO 的核心工程壮举：保留 RL 的主动探索，但彻底删除评论家网络。

替代方案：组内相对比较

不再依赖一个神经网络给绝对基线，而是对同一个 Prompt 并行采样一组完整推理路径，再在组内比较谁更好。

Prompt
给定同一个问题。
Group
并行采样 8 或 16 条回答。
Verifier
外部脚本给每条路径打分。
Mean / Std
计算组内均值和标准差。
Advantage
用相对分数替代 Critic 基线。

GRPO 要解决是PPO的一个痛点，就是Critic 在长链条推理任务上越来越不好用。

为什么？因为 Critic 的职责，是在模型生成到一半的时候，提前估计这一局最后大概能拿多少分。但在长 CoT，也就是长链式思维任务里，这件事非常难。

比如模型正在解一道很复杂的数学题，它刚写下第一行式子的时候，这一步到底是好是坏，往往没法立刻判断。因为后面几十步才会决定这条路径是否真的走通。于是，Critic 在早期 token 上给出的价值估计，天然就很不稳定。

而且前面也说了，Critic非常吃资源。所以GRPO就删除了它。

右边这部分是 GRPO 的替代方案，也就是组内相对比较。面对同一个 Prompt，不再只采样一条回答，而是并行采样一组，比如 8 条或者 16 条完整回答。然后，借助外部 verifier，也就是验证器，给这组回答逐个打分。接着，计算这一组样本的平均分和标准差。最后，每个回答的 Advantage 不再来自一个 Critic，而是来自它相对于同组其它样本的表现。

这一步非常关键。GRPO 不是问你绝对有多好，而是在问同一题下你是不是比你的同伴更好。这个思路一下子就把困难从学一个全局价值函数，变成了在一组候选里做相对排序。

GRPO 的关键数学：Z-Score 相对评分

公式很简单，统计意义却极强

$$A_i=\frac{R_i-\operatorname{mean}(R_{1..G})}{\operatorname{std}(R_{1..G})}$$

优势不再来自 Critic 的绝对价值估计，而是来自这条轨迹相对于同组平均水平的“拉开程度”。

低于均值

26
样本 C

类比：大学期末考试“拉曲线”

如果一场《实变函数》考试极难，全班平均只有 30 分，那么 50 分虽然仍不及格，但在班里已经是极强正样本；相反，若题目很简单，全班平均 90 分，那么 80 分反而是负优势。

  • 题目难度高低不再直接干扰梯度尺度。
  • 同组总能分出相对更好的推理路径，避免梯度失去方向。
  • 罕见但优秀的 CoT 会被 Z-Score 放大并固化。
“内卷式机制”：模型不是和题目难度打架，而是在自己当前能力边界上不断突围。

这一页是 GRPO 的核心。这个公式本质上就是 Z-Score，也就是一个样本的奖励，减去组内平均值，再除以组内标准差。

它最关键的意义是，模型更新不再依赖外部价值网络，而是直接用同组样本之间的相对位置来定义优势。

这和考试拉曲线很像。题目很难时，50 分可能已经明显领先；题目很简单时，80 分反而可能落后。GRPO 看重的不是绝对分数，而是当前组内的相对水平。

这样做有三个直接好处：不同题目难度带来的奖励尺度差异会被自动标准化；只要组内还能分出高下，训练就还有方向；少数高质量的长推理路径也更容易被放大。

GRPO + RLVR：用纯规则奖励点燃自我进化

DeepSeek-R1-Zero 的奖励系统极其朴素

准确性奖励
正确 = 1，错误 = 0
格式奖励
<think>...</think>

在数学与代码场景，很多时候几行验证脚本或形式化证明器就能替代黑盒神经奖励模型，这正是可验证奖励（RLVR）的威力。

结果：复杂推理行为从试错中涌现

  • Reflection：发现原路不通后主动换思路。
  • Self-correction：意识到错误后自发回滚重算。
  • Aha moment：长思维链中出现顿悟式跳跃。
这些能力并不是靠人工写标准答案“教”出来的，而是在规则奖励 + GRPO 相对评分下，通过强化学习自己涌现出来的。

GRPO 提供简单的学习信号。那奖励信号本身能不能也尽量简单？

在有些任务里，答案是可以的。左边这套 DeepSeek-R1-Zero 式奖励就很朴素，主要只有两类。第一类是准确性奖励，答对记 1，答错记 0。第二类是格式奖励，比如要求思考过程放进特定标签里，满足格式就加分。

它之所以可行，是因为数学题、代码题、形式化证明这类任务往往可以客观验证。我们可以跑测试、调用 verifier、做符号检查，因此不一定非要训练黑盒 Reward Model，而是可以直接使用可验证奖励，也就是 RLVR。

更重要的是，在GRPO和RLVR的结合下，复杂推理行为会自己涌现。模型会主动换路、回退重算，甚至在长推理中出现关键顿悟。这些能力不是工程师逐句教出来的，而是模型在大量试错中自己学会的。

真实的显存与算力账本

工业参考场景：8B 参数、20K 上下文。核心差异不是抽象优劣，而是显存账本能否落地。

PPO 的高成本来自 Actor、Critic、Reward、Reference 同时在场，还要维持优化器状态和大规模激活值。
GRPO 的革命性在于：去 Critic + 规则验证替代重奖励模型，再配合 LoRA 与激活检查点，显存可压缩约 90%。

前面讲的算法其实没有孰优孰劣。最关键的还是钱：你到底训不训得起。

算法优劣不应该只看论文曲线，还要看显存账本能不能落地。因为很多时候，实验室不是在问哪个算法理论上最好，而是在问我手里的硬件，到底允许我跑哪一种训练范式。

从落地角度看，如果你是资源非常充足的大实验室，PPO 仍然是一个强大而稳健的基线。如果你资源有限，但手里有高质量偏好数据，DPO 往往是性价比极高的选择。如果你做的是数学、代码、可验证推理，而且希望模型在训练中继续主动探索，那么 GRPO 会非常有吸引力。

前沿演进：DAPO / GSPO / ES

DAPO：修长度偏见

GRPO 会诱导模型通过“过短但不完整”的答案投机。DAPO 用不对称裁剪（Asymmetric Clip）和动态采样 / 截断屏蔽，逼模型靠真正的长逻辑链拿分，而不是靠偷步数。

GSPO：从 Token 提升到 Sequence

MoE 模型里，Token 级重要性采样方差很高。GSPO 把优化颗粒度抬到完整序列，更贴近“最终答案拿奖励”的本质，也更适配 MoE。

ES：无梯度黑盒路线

进化策略把模型看成黑箱：给权重注入高斯噪声，繁衍出一群子模型，只做前向评估，再聚合优胜变异方向更新。好处是几乎不吃反向传播显存，并且天然适合超大规模并行。

趋势不是追单一算法的绝对最优，而是围绕“任务类型 + 硬件边界 + 模型结构”寻找组合工程最优。

DPO与GRPO当然也不是终点。最近也涌现出DAPO / GSPO / ES等

未来不是都会收敛到同一个算法，而是会根据任务类型、模型结构和硬件边界，组合出不同的训练范式。换句话说，对齐算法的竞赛，越来越像系统工程，而不只是单点数学公式的竞赛。

总结：对齐——数学与工程的双重博弈

从将文本生成定义为马尔可夫决策过程，
到用 Bradley-Terry 模型量化人类偏好；
从 PPO 的稳健牵引、DPO 的优雅降配，
再到 GRPO 激发的自我思考浪潮——

强化学习正在深刻重塑大语言模型的认知边界。
奖励黑客的幽灵提醒我们：对齐不仅是数学与算力的竞赛，更是目标设定与漏洞防御的工程博弈。

最后总结一下我们今天的演讲。强化学习，正在深刻重塑大语言模型的认知边界。大语言模型的对齐，从来不只是让模型更听话这么简单。它本质上是在有限算力、有限数据和不完美奖励之间，不断寻找一个可用、可扩展、可控的平衡点。
